# Contributing

Thanks for improving this skill. One rule matters more than the rest here:

## Keep it anonymous — no real client, customer, or project specifics

This is a **public, general-purpose** skill. It's often built and refined while
doing real client work, so it's easy to let a real name or a project-specific
detail slip into an example. Don't. Before you open a PR, make sure **nothing you
add identifies a real person, client, employer, or project**:

- **No real names** — clients, customers, colleagues, end-users. Use roles ("the QA
  lead", "the reviewer") or neutral placeholders ("Example User", "Acme").
- **No project-identifying specifics** — internal product names, ticket IDs,
  client-domain jargon, distinctive feature names. Keep worked examples
  **domain-neutral** (a generic SaaS / CRUD / blog example carries the same lesson
  without the fingerprint).
- **Strip identifying data from screenshots, GIFs, and recordings** — names, PII,
  account numbers, tokens, and anything in a visible URL. Crop it out, or capture
  against **seeded / dummy data** instead of a real record.
- This applies everywhere a reader can see it: **code, comments, docs, example
  data, commit messages, and PR titles/bodies** — not just the obvious places.

A quick rule of thumb: if a reader could tell *whose* real work this came from,
it's not anonymous enough yet. Vague-but-neutral beats specific-but-identifying.

## Otherwise

- Keep changes focused and explain the *why* in the PR description.
- Match the surrounding style.
- Examples should teach the mechanism, not advertise a particular customer.
